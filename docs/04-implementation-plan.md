# 实施方案（WearableAgent Hub）

## 1. 实施目标

本实施方案的目标是把项目从“已完成搭建”推进到“可稳定演示、可明确验收”。

核心目标有三个：

1. 把真实 MIMO 接入明确落到配置、引擎、启动、验证四个层面
2. 把协议链路明确区分成“已实现”与“已真实联调”
3. 把文档、代码、演示方式整理成可交付状态

## 2. 实施原则

### 原则 1：配置优先
所有关键参数来自 `.env`，不在业务代码中硬编码。

### 原则 2：真实优先
默认链路应优先指向真实 API，而不是 mock。

### 原则 3：验证优先
每个关键模块都要给出可执行的验证方式。

### 原则 4：最小闭环优先
先保证最短链路稳定，再扩展复杂场景。

## 3. 实施阶段

### Phase 1：项目基线收口

目标：确保当前仓库已经具备清晰、统一、可运行的基线。

工作内容：

- 确认目录结构清晰
- 确认 `AGENTS.md` 能准确描述项目目标
- 确认 `docs/` 能覆盖目标、需求、架构、方案
- 确认 `.env` 与 `.env.example` 保持一致

验收标准：

- 新人可以通过文档快速理解项目
- 文档和代码描述保持一致

### Phase 2：真实模型接入固化

目标：让 MIMO 接入成为一个稳定、可验证的默认能力。

工作内容：

- 固化以下配置：
  - `OPENAI_API_KEY`
  - `OPENAI_BASE_URL`
  - `OPENAI_MODEL`
  - `PROVIDER_TYPE`
- 确认 `packages/core/src/config.py` 正确读取这些配置
- 确认 `packages/core/src/engine/agent_engine.py` 使用标准 `chat.completions.create`
- 确认 `packages/core/src/main.py` 启动日志包含：
  - `provider_type`
  - `model`
  - `base_url`

验收标准：

- 后端启动后，日志明确显示使用的是 MIMO 配置
- 核心推理链路可以通过真实 API 完成一次请求

### Phase 3：最短链路验证

目标：把“用户输入 → Agent 回复 → HUD 渲染”做成最稳定的核心演示链路。

工作内容：

- 确保眼镜 HUD 可连接后端
- 确保文字消息可通过 WebSocket 或 REST 推送到后端
- 确保后端能返回结构化结果
- 确保前端能渲染 A2UI 组件

验收标准：

- 用户输入后，HUD 能在短时间内展示结果
- 链路可重复演示
- 不依赖一次性手动配置

### Phase 4：A2A 链路验证

目标：确认主 Agent 能够真实调用子 Agent。

工作内容：

- 确保 translate-agent 正常启动
- 确保主 Agent 可检测翻译类意图
- 确保主 Agent 能通过 A2A 调用子 Agent
- 确保结果可以回到 HUD 展示

验收标准：

- 输入 `translate ...` 能触发 A2A 调用
- HUD 能展示翻译结果
- Dashboard/日志能看到调用链路

### Phase 5：x402 链路验证

目标：确认付费服务调用可以走完模拟支付闭环。

工作内容：

- 确保 x402 服务正常启动
- 确保主 Agent 能识别支付意图
- 确保调用链路能完成 pay-and-settle
- 确认前端能展示支付状态

验收标准：

- 触发付费场景后，系统能返回支付成功信息
- 支付记录可查

### Phase 6：多设备与观测能力验证

目标：确认眼镜、手表、Dashboard 能协同工作。

工作内容：

- 确保多前端同时连接同一后端
- 确保 Dashboard 能展示：
  - Agent 状态
  - 日志
  - 支付信息
- 确保手表端能接收到关键状态更新

验收标准：

- 多设备可同时运行
- Dashboard 可作为观测入口

## 4. 当前真实接入状态

### 已确认接入真实 MIMO

从代码和配置看，当前项目已经完成以下接入：

- `.env` 已配置 MIMO endpoint、key、model
- `.env.example` 保留了相同默认项
- `packages/core/src/config.py` 读取这些配置
- `packages/core/src/engine/agent_engine.py` 使用 `AsyncOpenAI` 调用 `chat.completions.create`
- `packages/core/src/main.py` 启动时打印关键配置信息

因此结论是：

> MIMO API 已经作为配置层真实接入。

### 尚需验证的部分

但“配置接入”不等于“全链路验收完成”。  
当前仍建议重点验证：

1. 真实 LLM 请求是否能在主链路中稳定返回
2. A2A 场景下是否会出现上下文拼接、超时、格式异常
3. x402 场景下是否与 LLM 结果正确串联
4. 语音模拟链路是否稳定

## 5. 重点实施建议

### 建议 1：优先稳定最短链路
先确保“输入 → 回复 → HUD 渲染”100% 可演示。

### 建议 2：再稳定协议链路
再依次验证 A2A、x402、多设备联动。

### 建议 3：把“配置层接入”升级为“验收级接入”
即除了 `.env` 之外，还要有：

- 启动验证
- 请求验证
- 错误提示验证
- 文档说明验证

### 建议 4：保持单机可运行
这是原型系统最重要的优势，不要过早引入复杂部署。
