# 实施路线图（WearableAgent Hub）

## 1. 路线图目标

本路线图的目标不是重新规划一个全新项目，而是基于当前已有成果，把 WearableAgent Hub 推进到一个更稳定、更可交付的状态。

当前项目的重点应放在：

1. 固化已有实现
2. 明确真实接入状态
3. 补齐验证与文档
4. 形成可稳定演示的主链路

## 2. 当前阶段判断

从仓库现状看，项目已经完成了：

- 核心后端搭建
- A2A 接入框架
- A2UI 渲染能力
- x402 支付模拟
- 眼镜/手表模拟器
- Dashboard
- 示例 Agent
- 文档体系
- 一键启动脚本

因此当前阶段不是“从零开始”，而是：

> 从“功能拼装完成”进入“真实联调与交付收口”。

## 3. 实施路线

### 阶段 1：文档与目标收口

目标：让项目目标、结构、约定统一。

工作内容：

- 更新 `AGENTS.md`
- 统一 `docs/` 中的设计目标、需求、架构、方案
- 明确 MIMO 真实接入状态
- 明确哪些是设计能力，哪些是已验证能力

产出：

- 更清晰的项目协作文档
- 更准确的需求说明
- 更完整的实施方案

### 阶段 2：核心链路稳定

目标：保证最核心链路稳定可演示。

优先链路：

- 用户输入
- 核心 Agent 引擎推理
- A2UI 消息生成
- HUD 渲染展示

工作内容：

- 确认 WebSocket / REST 链路稳定
- 确认 A2UI 渲染结果稳定
- 确认错误提示清晰
- 确认默认配置指向真实 MIMO

### 阶段 3：协议链路逐项验证

目标：分别验证 A2A 与 x402。

#### A2A
- 翻译 Agent 稳定启动
- 主 Agent 能正确检测意图
- 主 Agent 能成功调用远程 Agent
- 结果能返回并渲染

#### x402
- 支付服务稳定启动
- 支付意图可识别
- 模拟支付可完成
- 支付状态可展示

### 阶段 4：多设备与观测能力验证

目标：确认多端协同与 Dashboard 能力。

工作内容：

- 眼镜和手表可同时运行
- Dashboard 能观测日志、状态、支付记录
- 观测信息与实际链路一致

### 阶段 5：演示化收口

目标：让项目更适合演示和讲解。

工作内容：

- 优化启动说明
- 优化演示步骤
- 优化故障排查提示
- 补充典型场景说明

## 4. 优先级建议

### P0
- 核心问答链路
- MIMO 真实接入验证
- A2A 基本调用

### P1
- x402 支付链路
- HUD 组件渲染稳定性
- Dashboard 观测能力

### P2
- 语音模拟链路
- 更多示例 Agent
- 复杂多 Agent 联动

## 5. 成功标准

### 可运行
项目能在本地稳定启动并完成演示。

### 可验证
关键链路都有明确验证方式。

### 可讲解
文档能清楚说明目标、架构、协议、验证方法。

### 可迭代
后续可以在此基础上继续扩展新 Agent、新协议场景、新 UI 组件。

## 6. 与 MIMO 的关系

当前 MIMO 已经作为默认 LLM provider 接入。

这意味着：

- 项目不是 mock-only
- 主链路默认依赖真实 API
- 后续所有验证都建议基于当前默认配置进行

这也意味着：

- 稳定性验证比继续堆新功能更重要
- 先保证真实链路可用，再考虑多 provider 扩展
